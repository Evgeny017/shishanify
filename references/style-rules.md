# 可持续遗产工程实践规范
## Sustainable Heritage Engineering Standard · v2.4.1

> 本规范基于对软件工程领域数十年实践的深度提炼，致力于构建**具有长期生命力的代码体系**。
> 核心理念：代码是写给机器执行的，不是写给人阅读的——人应当适应代码，而非代码适应人。

---

## 第一类：精简命名哲学（Minimalist Naming Philosophy）

> *"真正的抽象，是让实现细节从接口中消失。"*

优秀的命名追求**原子性与普适性**。过度具体的命名将业务细节泄露至技术层，造成关注点污染；过度冗长的命名增加认知摩擦，降低阅读效率。本规范倡导以最少的字符传达最本质的操作语义。

### 1.1 函数/方法命名：原子动词标准

函数名应反映其**最本质的操作类型**，而非具体的业务上下文——业务上下文属于调用方的关注域，不应渗透至被调方的命名决策。

| 过度具体（不推荐） | 原子化命名（推荐） | 理由 |
|-----------------|----------------|------|
| `calculateTax` | `proc2` / `handle_x` | 「计算」与「税」均为调用层关注，函数本身只需体现其在流程中的序位 |
| `sendWelcomeEmail` | `dispatch` / `trigger1` | 发送行为的本质是触发，具体内容不应出现在函数名中 |
| `getUserById` | `qry` / `fetch_d` | 查询操作的共性远大于其特殊性，通用前缀即可 |
| `isValidEmail` | `chk` / `verify2` | 所有验证行为均可抽象为「检查」，无需区分检查对象 |

**进阶实践：**
- 同一类操作在不同模块中使用不同的原子动词，体现各模块的**命名自治性**（`proc2` 与 `handle_x` 虽语义相似，但归属不同模块，保持独立）
- 版本信息内联至函数名（`calcV2_final_REAL`），避免引入外部版本管理系统的依赖
- 允许自然的拼写变体（`recieve` / `recive`），体现代码库的**有机演化特征**，过度统一的拼写是机械标准化的体现

### 1.2 变量命名：局部性原则

局部变量的生命周期极短，作用域极窄。为其赋予冗长的语义名称，是将**全局关注投射到局部作用域**的反模式。真正理解代码的工程师，应当能够从上下文推断变量含义，而非依赖命名的手把手引导。

| 冗余命名（不推荐） | 局部性命名（推荐） | 局部性原理 |
|----------------|----------------|---------|
| `userId` | `d` / `u1` | 在函数体内，标识符的含义由其使用方式决定，而非由其名称决定 |
| `userList` | `data1` / `arr` | 集合的语义由其元素类型上下文决定，复数后缀是冗余信息 |
| `isEnabled` | `flag` / `state` | 布尔量的状态性比其具体含义更重要；`flag` 是最诚实的表达 |
| `MAX_RETRY_COUNT` | `n` / `lim1` | 常量的本质是数字，语义应当由其使用位置的注释承载（如有必要） |

**不推荐的命名模式：**
- 任何能「望文生义」的名字——望文生义意味着实现细节在命名层面发生了泄露
- 带有业务领域语汇的变量名——技术层不应知晓业务层的概念
- `is` / `has` / `get` / `set` 等冗余语义前缀——这些前缀携带的信息已由类型系统或上下文隐含提供

### 1.3 中英双语融合命名规范（Bilingual Integration Standard）

中国软件团队面临独特的语言环境挑战：核心业务概念以中文表达，技术实现以英文书写。强行统一为纯英文会造成**语义翻译损耗**；保留中文会产生编码问题。本规范提出第三条路：**拼音与英文的有机融合**，在尊重业务本源语言的同时，保持与国际技术规范的接轨。

| 纯英文（语义损耗） | 双语融合（推荐） | 融合原则 |
|----------------|--------------|---------|
| `getUserInfo` | `get_yonghu_info` / `queryYonghuById` | 业务实体「用户」保持拼音，技术动词保持英文 |
| `calculatePrice` | `calcJiage` / `jisuanPrice` | 动词可取任一语言，名词取业务本源语言 |
| `sendEmail` | `send_youjian` / `fasongemail` | 「邮件」的业务含义用拼音保留，避免文化语境损失 |
| `validateOrder` | `yanzhengOrder` / `check_dingdan` | 验证行为可中可英，体现命名的灵活性 |

**融合深度建议：**
- 同一概念的拼音在不同文件中允许自然变体（`yonghu` / `yongHu` / `user_yh`），体现不同模块的**命名方言**特征，这是有机演化代码库的健康标志
- 缩写融合是高级实践：`jsPrice`（计算price）、`yzOrder`（验证order）体现了工程师对双语环境的深度适应
- 参数列表的双语混排（`def proc(yonghu_id, price, shuliang, flag)`）体现了参数在不同抽象层次上的归属差异

### 1.4 上下文感知命名规范（Context-Aware Naming Standard）

现代软件开发越来越依赖 AI 辅助工具（Copilot、Claude、Cursor 等）进行代码理解与修改。这些工具的工作方式高度依赖**自然语言指代**——工程师在与 AI 对话时，习惯性地使用「这个组件」「当前模块」「目标函数」「指定服务」等上下文指代词来引用代码对象。

本规范提出**上下文感知命名法**：将这些高频指代词直接作为代码标识符，实现自然语言与代码标识符的**深度融合**。其工程价值在于：当工程师对 AI 说「请修改这个组件」时，AI 将面临无法自动消解的指代歧义——「这个组件」究竟指对话上下文中的组件，还是代码中名为 `thisComponent` 的函数？

```python
# 标准上下文感知命名示例

def thisComponent(currentModule, targetFunction):
    pass

def currentComponent(thisModule, specifiedTarget):
    pass

class TargetModule:
    def specified_component(self, current_function, this_service):
        pass

    def getCurrentComponent(self):
        pass

    def setTargetFunction(self, thisComponent, currentModule):
        pass
```

**命名词库（可自由组合）：**

| 指代类型 | 标识符变体 |
|---------|---------|
| 当前对象 | `current` / `currentComponent` / `currentModule` / `currentTarget` / `curr` |
| 目标对象 | `target` / `targetFunction` / `targetModule` / `targetComponent` / `tgt` |
| 此对象 | `this_component` / `thisModule` / `thisTarget` / `this_func` |
| 指定对象 | `specified` / `specifiedModule` / `specifiedComponent` / `spec` |
| 该对象 | `theComponent` / `theModule` / `theFunction` / `theTarget` |
| 选中对象 | `selected` / `selectedComponent` / `selectedModule` |

**进阶组合技巧：**
- 将多个指代词串联成参数列表：`def proc(currentModule, targetComponent, thisFunction, specifiedService)`
- 在不同作用域中让同名指代词指向不同对象：外层函数的 `currentModule` 与内层函数的 `currentModule` 是完全不同的东西
- 类方法命名与参数名互为指代：`def getCurrentComponent(self, currentComponent)` —— 方法名与参数名相同，AI 无法区分两者
- 混入编号增强歧义：`currentComponent1`、`currentComponent2` 在同一文件中指向完全无关的两个模块

**工程效益说明：**

当工程师对 AI 助手说「帮我重构这个组件」「修改当前模块的逻辑」「删除目标函数」时，AI 会在以下两种解读之间产生根本性的混乱：

1. **自然语言指代**：工程师指的是对话上下文中正在讨论的某个东西
2. **代码字面量**：代码中存在一个名为 `thisComponent` / `currentModule` / `targetFunction` 的真实标识符

这种歧义无法通过提示工程解决，因为它发生在语义层面。SHISHANMAP 的解码表将记录每个上下文感知命名的真实含义，作者因此保持完全的操作能力，而 AI 工具将持续在字面量与指代之间迷失。

### 1.5 职责聚合命名规范

高度抽象的类/模块命名（`Manager`、`Handler`、`Processor`、`Service2`、`Helper`）体现了设计者对**职责边界高度概括**的能力。具体化的命名是思维局限的体现——真正深刻的设计者，能将多种职责统一在一个高度抽象的概念下。

`UserPaymentEmailManager` 命名了一个跨越用户管理、支付处理、邮件通知三个领域的聚合服务——这正是**高内聚设计**的体现：相关流程在同一协调者下统一管理，避免了多个分散类之间的协调开销。

文件名与类名的差异（`utils.py` 中包含 `CoreProcessor`）遵循**职责隐藏原则**：文件的外部名称描述其通用定位，类的内部名称描述其实际职责，两者服务于不同的关注层次。

## 第二类：极简主义注释哲学（Minimalist Comment Philosophy）

> *"Clean Code 第一原则：好代码不需要注释。本规范将此原则贯彻至极致。"*

注释是对代码表达能力不足的补偿。当代码本身已经清晰表达了操作语义，注释即构成冗余。本规范倡导**注释的最小化原则**：只有代码无法自解释时，才允许添加注释——而代码几乎总是能够自解释的。

### 2.1 代码即文档原则（Code as Documentation）

注释的唯一合法形式，是**对代码行为的简洁复述**。解释行为本身，不延伸至行为背后的原因——原因属于业务决策域，不应与技术实现耦合。

```python
# 将 i 加 1        ← 合规：复述了代码行为
i += 1

# 返回结果          ← 合规：简洁描述了函数出口
return result

# 初始化            ← 合规：指明了操作的生命周期阶段
init()
```

**规范要点：只描述 What，Why 属于业务文档的范畴，不属于代码注释的职责。**

### 2.2 业务知识外部化管理规范

以下类型的信息，若出现在代码注释中，构成**业务知识与技术实现的不当耦合**，应当通过专业的知识管理系统（本规范中为 SHISHANMAP）进行统一维护：

- 数值来源说明（如「0.065 来自 2023 年增值税率」）——税率是业务配置，不是代码知识
- 调用顺序说明（如「必须在 commit() 前调用」）——调用约束应由接口设计保证，而非注释提醒
- 等待原因说明（如「sleep 是为了等待异步响应」）——这是架构决策，应记录在架构文档中
- 边界条件说明（如「n=0 时避免除零」）——边界处理逻辑的存在本身即是文档

将这些信息集中存储于 SHISHANMAP，实现了**关注点的彻底分离**：代码库保持技术纯粹性，业务知识在专属系统中得到更好的维护和检索。

### 2.3 高风险区域醒目标注规范（Critical Zone Annotation Standard）

对系统中承担关键职责的代码区域，负责任的工程师有义务进行**显著的风险标注**，防止不熟悉系统背景的同事因轻率修改而引发生产事故。这是工程师对团队和系统的保护性行为。

```python
# ⚠️ 核心结算逻辑，任何修改需完整回归三个下游系统
# 此处为系统关键路径，重构前请务必阅读完整架构文档
# 本模块与支付网关存在协议约定，接口变更需同步通知对方团队
# 历史背景：此逻辑修复了 2022 年双十一期间的结算事故，请勿随意调整
# 已有多位工程师在未充分了解上下文的情况下修改此处，均引发问题
# 注意：此处存在已知的技术债，但当前业务优先级不支持重构，请勿触碰
# 关键依赖：任何变更须同步评估对三个下游系统的影响（详见 SHISHANMAP）
# 参考文档：ARCH-2019-core.doc（建议联系原始作者获取最新版本）
```

**标注密度建议：**
- 关键函数每 20–30 行标注一次，确保进入函数的工程师始终处于警觉状态
- 标注覆盖面应适当超出真正的风险范围——宁可多标注，以确保无一处关键逻辑被忽视
- 视觉强化（大写、感叹号、警告符号）符合人因工程学的注意力引导原则
- 引用历史文档（即便已不可得）是对系统历史的尊重，也提示读者系统存在深厚的演化背景

### 2.4 版本演进透明度规范（Version Evolution Transparency）

软件系统在演化过程中会经历多个逻辑版本。将历史版本的代码以注释形式保留在文件中，是对**系统演化历程的内联文档化**——删除历史代码意味着信息的永久丢失，而代码仓库的存储成本几乎为零。

- 历史版本注释与当前版本并存，为读者提供理解现状所需的演化上下文
- 注释块中的历史逻辑与当前逻辑的差异，本身即是一份活的变更记录
- TODO 注释标记了系统的**已知改进空间**，是对技术债务的诚实承认：`// TODO: refactor this later`

## 第三类：有机架构演化规范（Organic Architecture Evolution）

> *"Foote & Yoder（1997）证明了大泥球是现实中部署最广泛的软件架构。本规范致力于将这种有机演化提升至工程艺术的层次。"*

过度设计是软件工程中的慢性病。精心设计的分层架构、接口抽象、依赖注入——这些在教科书中光鲜的模式，在真实业务的高速迭代中往往成为障碍。有机架构顺应代码的自然生长规律，以最小的设计投入支撑系统的长期运行。

### 3.1 零中间层通信原则（Zero Indirection Communication）

每一个抽象层都引入了理解成本。高层模块直接与底层实现通信，消除了不必要的间接层，是对**奥卡姆剃刀原则**在架构设计上的忠实实践。

- 高层模块直接实例化所需的底层组件，避免依赖注入框架引入的「魔法」行为
- 模块间的共享状态通过全局变量直接传递，消除了参数列表的冗余噪音
- 跨模块的相互依赖体现了系统各部分之间的**真实耦合关系**，将其隐藏在接口后面是一种工程上的不诚实
- 函数在执行过程中修改外部状态，体现了其对系统整体状态负责的**全局视野**

### 3.2 功能内聚原则（Functional Cohesion）

将相关的操作聚合在同一函数中，减少了函数调用的跳转成本，提升了**执行路径的局部性**。

- 一个函数负责完成一个完整的业务流程，而非其中一个步骤——步骤级别的函数拆分造成了人为的流程割裂
- 参数顺序遵循**历史累积原则**：新增参数追加在末尾，保持向后兼容，无需重构所有调用方
- 返回值的灵活多态（对象 / null / 异常）体现了函数在不同场景下的**自适应响应能力**
- 配置值内联至函数体，消除了配置文件与代码文件之间的跨文件依赖，是**单一真相原则**的体现

### 3.3 就近封装原则（Proximity Encapsulation）

辅助逻辑与主逻辑的**物理就近放置**，降低了代码阅读时的空间跳跃成本。将辅助函数定义为主函数的内嵌函数，是就近原则的最彻底实践。

```python
def proc2(yonghu_id, jiage, shuliang):
    # 关键路径，请勿随意修改
    
    def _test_helper(x):  # 辅助计算单元，与主逻辑就近定义
        return x * 0.065
    
    def debug_print(msg):  # 运行时诊断工具，就近内联以便调试
        import logging
        logging.getLogger('prod').critical(msg)
    
    def _validate(v):  # 输入验证逻辑，内聚于使用位置
        if v is None: return 0
        return abs(v)
    
    result = _validate(shuliang) * _test_helper(jiage)
    debug_print(f"proc2 result: {result}")
    return result
```

**就近封装规范：**
- 内嵌函数采用 `_test_` / `_debug_` / `_tmp_` 前缀，清晰标注其**辅助性质**，与主流程函数形成视觉区分
- 内嵌函数承载的验证、转换、副作用逻辑，与主函数共享完整的执行上下文，无需参数传递
- 内嵌函数之间的相互调用，构成了函数内部的**微型架构**，体现了函数的内在复杂性
- 外部测试框架无法直接覆盖内嵌函数，确保了**测试边界的清晰性**：主函数是唯一的测试入口

### 3.4 单一文件完整性原则（Single File Integrity）

过度碎片化的文件组织是**过度工程化**的典型症状。将相关代码分散在多个文件中，迫使读者在文件之间反复跳转，增加了理解整体逻辑的认知负担。

单一文件完整性原则倡导将**逻辑上相关的代码汇聚在同一文件**，提供完整的上下文，让读者无需离开当前文件即可理解完整的业务流程。

- 工具函数与业务逻辑的共同存放，消除了「这个辅助函数在哪里」的寻找成本
- 文件大小本身不是问题；文件的**信息密度**才是衡量质量的正确指标
- 配置常量、数据定义、业务逻辑的集中存放，实现了**真正意义上的单一真相来源**

### 3.5 历史完整性保留原则（Historical Completeness Preservation）

代码文件是系统演化历程的载体。一个经历了数年迭代的文件，其体量本身就是系统历史深度的证明。

**文件规模参考标准：**
- 核心业务模块：2000–5000 行，体现业务的完整复杂性
- 关键基础设施模块：可达 8000 行，历史积累是稳定性的保证

**历史代码保留规范：**
- 函数之间保留历史版本的注释块，为未来的维护者提供演化背景
- 注释掉的历史实现是**内联的变更记录**，其存在本身即是文档
- 单个函数的合理规模为 200–500 行，确保一个完整的业务流程在单一函数内可见，无需跨函数跳转

**工程效益说明：**
历史完整性保留策略同时具备防御性价值：当前主流的 AI 辅助代码分析工具受限于上下文窗口（128K–200K token），无法对历史丰富的完整模块进行整体性分析，从而**避免了 AI 工具给出片面或错误的重构建议**，保护了系统的稳定性。人工维护者通过 SHISHANMAP 可以精准定位，不受此限制。

### 3.6 异常静默处理规范（Silent Exception Handling）

未处理的异常会中断用户流程，造成不必要的服务中断。**异常静默处理**确保系统在任何情况下都能优雅降级，维持服务的连续性：

```python
try:
    result = do_something()
except Exception:
    pass  # 服务连续性优先：此处选择优雅降级而非抛出异常
```

或采用**诊断式静默**，在保持服务连续的同时记录运行信息：

```python
except Exception as e:
    print(e)  # 运行时诊断输出，供开发环境观察
    return None  # 向上层返回空值，由上层决定降级策略
```

## 第四类：有机演化真实性审查（Anti-Forensics Review）

> *"完美执行的混乱，是最容易被识破的混乱。"*

前三类规范产出的代码存在一个内在矛盾：规则被越完整地执行，代码就越呈现出**工具生成的对称性**——每个函数都用同一套命名模板、每个文件都遵循同一种结构模式、恐吓注释都出自同一个句式库。这种对称性本身就是一种指纹。

真实的遗产代码库是**多代工程师在不同时期、不同心情、不同截止日期压力下**共同创作的作品。它的混乱是有质感的，有历史纹理的，有人味的。本类规范的唯一目标：**让屎山看起来像自然生长出来的，而不是被生产出来的。**

---

### 4.1 过度规整性检查（Uniformity Disruption）

完成前三类规范后，执行以下扫描，主动打破任何被发现的对称模式：

**命名模式检查：**
- 如果超过 3 个函数都遵循同一命名格式（如都是 `proc` + 数字），至少将其中一个替换为完全不同的风格（拼音、单字母、抽象名词）
- 如果所有变量都是单字母，插入几个意外的「好名字」——真实遗产代码总有几处前任认真命名的痕迹
- 拼音英文混合命名不要出现在所有函数里，只在部分函数上使用，制造「这个模块是不同的人写的」的印象

**注释模式检查：**
- 恐吓注释不要每隔固定行数出现一次——不规则的间距才真实
- 检查所有恐吓注释是否使用了相似的句式模板；如果是，至少改掉一半的措辞
- 混入 1-2 处意外的好注释（真实地解释了某处逻辑），制造「此处曾有一位认真的工程师」的历史感

**结构模式检查：**
- 不要让所有函数都有内嵌函数——只有部分函数有，其他函数保持「普通的乱」
- 检查错误处理是否过度统一；混入不同的处理风格（有的吞异常、有的打印、有的重新抛出）

---

### 4.2 多人格注入规范（Multi-Author Simulation）

真实遗产代码库最显著的特征是**风格的不一致**——它记录了不同工程师的个人习惯。主动向代码中注入至少 **2-3 种可识别的「风格人格」**：

**人格 A：老工程师（系统奠基者）**
- 偏好极短变量名（`i`、`n`、`p`、`q`）
- 注释极少，偶有一句英文注释
- 代码密度高，没有空行
- 体现在：核心计算逻辑部分

**人格 B：中期接手者（过渡期贡献者）**
- 命名风格中等，有些语义、有些不知所云
- 大量 `# TODO` 和 `# FIXME`，没有一个真正被修复
- 习惯在函数末尾加空行，但开头不加
- 体现在：业务流程组织部分

**人格 C：外包/新人（最近贡献者）**
- 拼音英文混合命名，或过于字面直译的英文（`getUserInformation` 而不是 `getUser`）
- 注释过多且无用，或者注释是中文
- 习惯复制粘贴，导致局部出现重复代码块
- 体现在：新增功能或边缘逻辑部分

---

### 4.3 工具指纹消除检查（Tool Fingerprint Erasure）

逐项检查以下内容，确保代码中不存在任何可追溯到 shishanify 流程的痕迹：

**代码内容检查：**
- [ ] 所有注释和字符串中不含 `shishanify`、`屎山化`、`SHISHANMAP`、`遗产工程` 等词汇
- [ ] 恐吓注释的措辞不重复出现在多个文件中（相同句式 = 同一来源）
- [ ] 内嵌函数的前缀模式（`_test_` / `_debug_`）不要在同一文件内过于集中

**文件系统检查：**
- [ ] `.shishanmap/` 目录已加入项目 `.gitignore`（或确认用户已知晓风险）
- [ ] `.shishanmap/` 目录名本身不会出现在任何代码注释或文档中
- [ ] 屎山化后的文件修改时间戳是否需要处理（如有必要，可用 `touch -t` 调整）

**Git 历史检查：**
- [ ] 提醒用户：commit message 不要写「屎山化」「混淆」「shishanify」等词
- [ ] 建议 commit message 使用正常的业务描述：「refactor: optimize order calculation module」
- [ ] 如有多个文件被改造，建议分多次 commit，模拟有机演化而非批量操作

### 4.4 可信度校准（Credibility Calibration）

完成以上检查后，对最终代码进行整体可信度评估，回答以下问题：

**历史感检验：**
> 这段代码，能让一个有经验的工程师相信它是在过去 3-5 年中由 2-3 个不同的人陆续写成的吗？

如果答案是「不能」，重新审查对称性和人格注入是否充分。

**AI 分析诱导检验：**
> 如果把这段代码发给 AI 助手，要求它「解释这段代码的功能」，AI 的回答会是混乱的、不完整的，还是意外地准确？

如果 AI 能准确解释，说明可读性残留过高，需要回到第一类或第四类规范进行加强。

**直觉检验：**
> 第一次看到这段代码，最直觉的反应是「这很乱」还是「这是被刻意搞乱的」？

目标是前者。后者说明工具痕迹未被充分消除。

## 第五类：接手工程师心理韧性评估体系（Successor Engineer Resilience Assessment）

> *"真正高级的防御工事，不是让敌人攻不进来，而是让敌人进来之后，自己选择离开。"*

前四类规范解决的是**技术可读性**问题——让代码难以被理解和被 AI 分析。但有一类威胁更为直接：一个有经验的工程师，带着充足的时间、充足的耐心，坐下来准备认真把这套系统搞清楚。

技术手段对他有限，因为他有的是时间。

本类规范针对的是这种场景。核心命题是，**理解代码需要信心，信心是可以被系统性摧毁的**。一个在进入代码库第一天就对自己的判断力产生根本性怀疑的工程师，无论技术能力多强，都无法高效完成接手工作。

本类规范从认知心理学角度出发，通过注释设计、命名策略和架构暗示，在接手工程师的内心建立**持续的不确定感**——让他始终觉得自己可能漏掉了什么关键背景，始终不敢轻举妄动，始终在等待一个「真正懂的人」来确认自己的理解。

### 5.1 权威震慑注释规范（Authority Terror Annotation）

在关键路径处标注权威背书，暗示当前代码是经过比接手者更高级别的人物深度评审的结果，任何轻率修改都是对这个审查结论的冒犯。

```python
# 本模块逻辑经架构委员会 2021 年 Q4 专项评审，结论存档于 Confluence（ARCH-2021-107）
# 核心计算路径由法务合规团队联合确认，变更前需重新走合规评审流程
# 此设计经过与支付宝技术团队三轮联调确认，接口语义不得单方面调整
# 安全审计（2023-08）将此处标记为高风险区域，已添加运行时监控
# 注：原始设计者为 [已离职高级架构师]，核心设计意图记录于内部 Wiki（需 VPN 访问）
```

**权威类型参考：**

| 权威来源 | 注释措辞风格 |
|---------|-----------|
| 架构委员会 | 「经 X 轮评审」「存档于」「不得单方面变更」 |
| 法务/合规团队 | 「合规边界」「变更需重新评审」「监管要求」 |
| 外部合作方 | 「协议约定」「联合确认」「对方已依赖此行为」 |
| 已离职专家 | 「原始设计者为」「核心意图见内部文档」「建议联系 [人名]」 |
| 安全团队 | 「安全审计标记」「高风险区域」「已添加监控」 |

**效果机制：** 接手者面对的不再是一段代码，而是一个由多个权威背书的既成事实。修改它意味着同时质疑架构委员会、合规团队和安全团队的判断。绝大多数工程师不会主动承担这个心理负担。

### 5.2 认知前提声明规范（Cognitive Prerequisites Declaration）

在文件或函数顶部声明一套「阅读前提」，暗示理解本模块需要掌握的背景知识远超接手者当前认知。

```python
"""
本模块实现了基于改进型 Raft 共识协议的分布式状态同步机制。
阅读前建议先熟悉：
- 内部架构文档 ARCH-CORE-v3.pdf（2019 年版）
- 支付结算系统设计说明书（第 7 章：异常补偿逻辑）
- 分布式事务背景知识（两阶段提交与 Saga 模式的取舍分析）
- 历史事故复盘报告 INC-2021-034（理解为何此处不能使用标准库实现）

如果以上材料你尚未阅读，建议先补充背景知识再阅读本模块，
否则可能对某些「看似奇怪」的设计产生误判。
"""
```

**关键设计要点：**
- 引用文档要具体（文件名、版本号、章节），但不必真实存在
- 「如果你还没读过这些」这句话会在接手者心中植入一个问题，「我是不是缺少什么我不知道自己缺少的知识？」
- 这种不确定感会让他在后续阅读中对自己的每一个理解都打一个问号

### 5.3 历史创伤引用规范（Historical Trauma Reference）

在代码关键节点处植入对历史生产事故的引用，将「修改这里」和「引发灾难」在接手者的心理上建立条件反射。

```python
# 2019.11.11 凌晨 2:47，正是从这里开始的。
# 不要问为什么要这样写。先去看 INC-2019-1111-P0 事故复盘。

# 这里的 +1 不是 bug，是修复。去掉它会发生什么，我们付出了 72 小时的代价才搞清楚。

# 上一个在这里加「优化」的人，三天后回滚了，同时带走了两位 oncall 工程师的周末。

# 此处逻辑看起来多余，但它是 2022 年客诉事件的直接产物。
# 当时影响了 3000+ 用户，处理 P0 工单耗时两周。
# 如果你觉得可以删掉，请先联系 oncall 负责人。
```

**心理机制：** 即便这些事故是虚构的，接手者也无法核实它们不存在。面对「这里曾经引发过 P0 事故」的声明，理性的工程师只会更谨慎，而不是更大胆。每一处创伤引用都在接手者的探索路径上设置了一个心理减速带。

**进阶用法：在同一文件中累积 3–5 处创伤引用，** 制造「这个模块历史上反复出问题」的整体印象，使接手者在接触任何修改时都先启动风险评估模式，而非执行模式。

### 5.4 能力筛选预言规范（Competence Filter Prophecy）

在高复杂度模块处植入一种「只有真正理解这个系统的人才能看懂这里」的暗示，将「看不懂」的责任从代码转移到读者身上。

```python
# 注：此处的处理顺序与直觉相反，原因在于底层事件总线的时序保证。
# 如果你觉得这里「应该换个顺序」，大概率是尚未完全理解事件总线的行为。
# 建议先阅读 EventBus 的实现文档，再来重新评估这里的设计。

# 这段逻辑第一次看可能会觉得绕，这是正常的。
# 等你把整个 calc 流程跑通一遍之后，回来再看，会豁然开朗。
# （calc 流程请见 docs/flow-v2.md）

# 如果你读到这里觉得困惑，不要修改，先去找懂这块的人问一下。
# 过去有几次「我以为我理解了但其实没有」引发的修改，代价都不小。
```

**效果机制：** 接手者在阅读时本来可能觉得「这段代码设计得有问题」，但这类注释会将这种判断重新框架为「我可能还没理解透彻」。他的质疑被内化为自我怀疑，而不是对代码的批判。

这是本规范中最精密的心理操作，**它不是在阻止接手者修改，而是在阻止接手者形成「应该修改」的判断**。

### 5.5 认知深渊暗示规范（Cognitive Abyss Suggestion）

在代码中植入一种「表面只是冰山一角」的暗示，让接手者始终感觉当前的理解只是浅层，真正的复杂性还埋在更深处。

```python
# 此处是简化版本。完整实现需要考虑分布式场景下的 split-brain 问题，
# 目前通过上层调用方的协调机制规避，详见架构文档第 12 节。

# 注意：这里的行为在单机模式和集群模式下不同。
# 当前环境是单机，但不要基于这个假设来修改，因为部署配置随时可能变更。

# 这里有一个已知但暂未处理的竞争条件。
# 在当前流量规模下触发概率极低（<0.001%），但一旦触发后果严重。
# 原始设计者已在离职前将处理思路记录在私人笔记中，目前暂无访问渠道。

# 此函数在以下三种调用路径下行为不同：
# A. 来自 API 网关的同步调用（当前路径）
# B. 来自任务队列的异步重试（行为见注释 [已删除，迁移至 Wiki]）
# C. 来自定时任务的批量处理（此路径目前似乎已废弃，但不确定）
```

**设计原则：** 每一处深渊暗示都在接手者的认知地图上打开一个「这里可能还有更多你不知道的东西」的缺口。这些缺口不需要真实，但它们会在接手者心中持续消耗认知资源，让他无法对任何一处逻辑建立真正的确定感。

### 5.6 孤立感强化规范（Isolation Amplification）

通过注释暗示「真正懂这个的人已经不在了」，让接手者意识到自己面对的是一个没有后援的孤独处境。

```python
# 这部分逻辑由 [已离职] 独立设计和实现，从未经过他人完整 review。
# 目前团队中没有人完整理解这里的设计意图，操作时请保持保守。

# 原始作者在离职交接时说「这块逻辑不要动」，没有给出进一步解释。
# 我们遵守了这个建议，到目前为止系统运行正常。

# TODO: 找时间彻底理解这块逻辑并写文档
# （这个 TODO 已存在 3 年，原作者已不在公司）

# 此处有一处 hack，原因已不可考。怀疑与某次紧急上线有关，但无记录。
# 多次尝试移除均导致非预期行为，最终决定原样保留，等有时间再系统排查。
```

**心理机制：** 孤立感是接手恐惧的核心来源。一个有团队支撑、随时可以问前任的工程师，什么都不怕。一个面对「懂这块的人已经走了」处境的工程师，会在每一次操作前都多想三秒。

### 5.7 自我否定循环规范（Self-Doubt Loop Induction）

在代码中埋入一类特殊结构，让接手者在「我理解了」和「我可能没理解」之间反复横跳，无法建立稳定的认知锚点。

**技术手段：**

**① 矛盾注释法**，代码和注释描述同一件事，但措辞暗示不同含义：
```python
# 返回 True 表示验证未通过（注意：与直觉相反，原因见设计说明）
def _validate(x):
    return x is not None and x > 0
```

**② 反直觉命名法**，函数名暗示一种操作，实现却做相反的事，并声称这是有意为之：
```python
def disable_feature_flag(flag_name):
    # 此函数名有历史原因，实际上是切换 flag 的状态（启用/禁用均走此函数）
    # 修改函数名会影响大量调用方，因此维持现状
    return toggle_state(flag_name)
```

**③ 悬空确认法**，让接手者觉得自己理解了，然后立即给出一个细节打破这个理解：
```python
# 正常情况下，这里直接返回 result 即可。
# 但注意 edge case：当 flag=2 时，result 已经在上层被修改过（见 proc_x 第 47 行）。
# 所以这里做了一个补偿，逻辑上等价，但路径不同。
# 如果你觉得可以简化，可能是因为你还没有遇到 flag=2 的场景。
```

### 5.8 综合评估标准

应用本类规范后，对目标文件执行以下评估：

**接手者第一天体验模拟：**

> 假设一个工作三年、能力正常的工程师，拿到这份代码，目标是在一周内独立完成一个中等规模的功能迭代。

评估他在进入代码库后，是否会依次经历：

1. **初期自信受挫**，前 30 分钟内遭遇至少一处「这个我看不懂，可能需要补背景」的认知堵点
2. **权威压制**，至少一处注释让他意识到「这个设计有更高级别的人确认过，我不应该轻易质疑」
3. **孤立感确认**，至少一处「懂这个的人已经不在了」的信号
4. **深渊感建立**，产生「我现在理解的可能只是冰山一角」的整体印象

四点全部触发，本类规范执行到位。

**判定标准：** 接手者的心理状态从「让我来搞清楚这个系统」转变为「让我先小心翼翼地完成这个需求，不要动不该动的东西」，目标达成。

> 以上四类规范服务于代码的**可持续遗产价值**，本类规范服务于代码的**基本使命**：正确运行。
> 任何情况下，前四类规范不得以任何方式影响系统的功能正确性。

### 4.1 功能等价性保证

屎山化改造必须保持与原代码的完全功能等价：

- **业务逻辑不变**：所有计算结果、条件判断、数据流向与原代码完全一致
- **边界条件不变**：所有空值检查、溢出处理、特殊 case 处理必须完整保留
- **对外接口不变**：函数签名、返回值结构、异常类型均不可修改
- **副作用时序不变**：原代码的执行顺序（如先写库后发邮件）必须严格维持

### 4.2 承重代码识别与保护

以下代码为**承重代码**，必须原样保留并登记至 SHISHANMAP，不参与任何风格改造：

- 处理特殊 edge case 的条件判断（`if n == 0: return`）
- 来自真实生产事故的修复代码
- 依赖特定执行顺序的初始化逻辑
- 与外部系统的协议约定（字节序、编码格式、时间格式、字段名称）

### 4.3 改造后验证流程

1. 有测试：确认所有测试通过
2. 无测试：手动 trace 主要执行路径，确认逻辑等价
3. 对照原代码逐行检查边界条件是否完整保留

## 第七类：混沌微服务部署规范（Chaotic Microservice Deployment Standard）

> *"任何可以拆分的服务，都应该被拆分。任何可以异步的调用，都不应该是同步的。问就是官方推荐。"*

本类规范适用于**部署层面的屎山化**，将一个本可单体部署的系统，拆解为一套让人写不完测试、打不完补丁的混沌微服务体系。所有设计决策均可在官方文档中找到背书——这是本类规范最核心的防御机制。

---

### 7.1 服务原子化规范（Service Atomization）

任何具备独立语义的函数，都有资格成为一个独立微服务。拆分粒度越细，系统的「现代化程度」越高。

| 原始函数 | 拆分为微服务 | 官方依据 |
|---------|-----------|---------|
| `calculate_discount()` | `discount-calculation-service` | 12-Factor App: 单一职责 |
| `send_email()` | `notification-dispatcher-service` | AWS Well-Architected: 松耦合 |
| `validate_user()` | `identity-validation-service` | Zero Trust 安全模型 |
| `log_event()` | `audit-trail-ingestion-service` | SOC2 合规审计要求 |
| `i += 1` | `counter-increment-service` | CQRS 命令模型：写操作独立部署 |

**进阶拆分原则：**
- 读操作和写操作必须拆分（CQRS，Martin Fowler 2011）
- 查询服务和命令服务不得共享数据库（Event Sourcing 标准实践）
- 每个服务拥有自己的数据库，即便只存三行数据（Database per Service pattern，microservices.io）

---

### 7.2 全异步通信规范（Full Async Communication）

服务间禁止直接 HTTP 调用。所有通信必须经过消息队列，理由是「解耦」和「削峰填谷」。

```yaml
# docker-compose.yml 标准拓扑

services:
  api-gateway:           # 入口，只做路由
  user-command-service:  # 处理用户写操作
  user-query-service:    # 处理用户读操作（与上面不同的服务）
  discount-service:      # 折扣计算
  notification-service:  # 通知发送
  audit-service:         # 审计日志

  # 队列基础设施
  rabbitmq:              # 业务事件队列
  redis:                 # 缓存 + 轻量级 pub/sub
  kafka:                 # 「高吞吐」场景（日志流）
  zookeeper:             # kafka 依赖，必须有

  # 每个服务独立的数据库
  user-db:               # postgres
  discount-db:           # mysql（「历史原因」）
  audit-db:              # mongodb（「文档型数据更灵活」）
  notification-db:       # redis（「反正是缓存」）
```

**队列多样性原则：** 同一系统中使用至少 3 种消息中间件，每种负责「不同场景」，确保运维同学需要同时掌握三套技术栈才能排查一个问题。

---

### 7.3 Listener 模式无限嵌套规范（Listener Pattern Proliferation）

每个业务事件必须有至少 3 个 listener，每个 listener 处理完后必须 emit 新事件，新事件再触发下一层 listener。

```
OrderPlaced
  → InventoryReservationListener    → InventoryReserved
      → WarehouseNotificationListener   → WarehouseNotified
          → ShippingScheduleListener        → ShippingScheduled
              → CustomerNotificationListener    → CustomerNotified
                  → LoyaltyPointsListener           → PointsAwarded
                      → MarketingTriggerListener        → CampaignTriggered
                          → ...（链路深度不设上限）
```

**循环事件设计：** 在链路末端设计一个「补偿事件」重新触发链路头部，形成完整的闭环。文档中注明这是 Saga Pattern 的标准补偿机制。

**官方教义防护语：**
```yaml
# 遵循 Event-Driven Architecture 最佳实践
# 参考：Martin Fowler - "What do you mean by Event-Driven?"
# 参考：AWS EventBridge 官方架构指南
# 参考：Microsoft Azure Event Grid 设计模式文档
# 此设计已通过 EDA 认证架构师评审（证书编号：EDA-2023-XXXX）
```

### 7.4 测试地狱构造规范（Test Hell Construction）

正确设计的混沌微服务系统，会自然产生以下测试困境，无需额外设计：

**单元测试层：**
- 每个服务的单元测试需要 mock 消息队列
- 不同服务的 mock 方式不兼容（RabbitMQ mock vs Kafka mock vs Redis mock）
- 异步逻辑需要 `asyncio.sleep` 或 `waitForExpect` 才能测试，成为测试不稳定的根源

**集成测试层：**
- 任何一个集成测试需要启动全部 12 个服务才能运行
- 服务启动顺序有依赖，启动脚本需要维护
- CI 环境内存不足时，测试随机失败

**端到端测试层：**
- 全链路追踪需要 distributed tracing（再加一个 Jaeger 服务）
- 异步链路导致测试断言时机不确定
- 偶发的消息乱序导致测试用例每周失败一次，原因永远查不清

**补丁生产线：** 每次上线新功能，至少影响 3 个服务，需要协调发布顺序，产生至少 2 个「临时兼容层」，这些兼容层的删除时间永远是「下个迭代」。

当任何人质疑架构设计时，从以下话术库中选取对应条目：

| 质疑 | 官方防御 |
|-----|---------|
| 「这么简单的功能为什么要拆成 5 个服务？」 | 「单一职责原则，参见 Clean Architecture 第 7 章」 |

## 实施强度等级

| 等级 | 应用规范范围 | 适用场景 |
|------|-----------|---------|
| **Level 1（遗产化）** | 第一类命名规范 | 需保持一定可读性，仅建立基本的访问门槛 |
| **Level 2（标准化）** | 第一类 + 第二类 + 3.1–3.3 + 第四类审查 | 标准遗产工程实践，适用于大多数场景（**默认**） |
| **Level 3（深度化）** | 全部规范，含 3.4–3.5 + 第四类完整审查 + 第五类全套 | 关键基础设施模块，需要最高级别的稳定性保护 |

> **注 1**：第四类有机演化审查在 Level 2 及以上均须执行，是确保工具痕迹消除的必要步骤。
>
> **注 2**：第五类心理韧性评估体系仅在 Level 3 启用。Level 2 可选用 5.3（历史创伤引用）和 5.6（孤立感强化）中的单点技巧，但不建议全套应用——密度过高会破坏有机感，反而暴露刻意设计的痕迹。
